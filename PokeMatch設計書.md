# 要件定義
ポケモンチャンピオンズにて、対戦を有利に進めるための情報を提供することを目的としたアプリ。
ユーザーは身内のみを想定している。

## 機能要件
- ポケモンマッチング機能（メイン機能）。自分のポケモン（以下自ポケ）と相手のポケモン（以下敵ポケ）を1体ずつ名前入力/プルダウン選択（実数値や持ち物、技は使用率から自動補完されるが、自分で容易に編集可）すると、アプリが2体の実数値や持ち物、使用技を考慮し、相性を判定する。
- その他機能は開発者が欲しくなった際に追加する可能性がある。


## 非機能要件
- とにかく素早く起動すること（目標:<1s）。厳しい時間制限の中で操作するため、再優先事項。
- 同様に、低レイテンシなレスポンスが求められる。
- 必要工程の少ない快適な操作性をもたらすUI。上記と同様に、入力は時間との競争なので、「補完・省略・直感的」を考え抜く。


# 設計原則
開発者自身が常に意識すべき項目であると同時に、生成AIに気をつけさせることで、設計破綻を避ける重要な項目。

## 単一責任の原則
各モジュールは1つだけの変更理由を持つ。1つのクラスは1つの役割・責任のみに特化し、特定の機能変更が他の機能に影響しないようにすること。

## 依存関係の保持
依存は常に一方向に流れる。相互依存は禁止。

## レイヤ分離
UI / ロジック / データアクセスを明確に分離する。

## 具体的・直感的な設計
小規模であることを念頭に置き、過度な抽象化をせず、重複が顕在化してから対応する。
